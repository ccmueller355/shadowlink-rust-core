# Implementation Plan: CLI Integration

**Branch**: `002-cli-integration` | **Date**: 2026-06-14 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/002-cli-integration/spec.md`

## Summary

A clap-based Rust CLI binary (`shadowlink-cli`) that exercises every capability of `shadowlink-rust-core` against real Matrix homeservers. The CLI is a developer/tester tool — not a user-facing application — consumed as a standalone binary in a separate public repository. Already fully implemented (382 lines, 11 commands). This plan documents the completed implementation and remaining verification gaps.

## Technical Context

**Language/Version**: Rust 2024 edition

**Primary Dependencies**: `shadowlink-rust-core` (git dep, `main` branch), `clap` 4 (derive), `tokio` 1 (rt-multi-thread, signal), `tracing` 0.1

**Storage**: `shadowlink_data/session.json` + `shadowlink_data/store/` (SQLite, managed by core's matrix-sdk) — CWD-relative

**Testing**: `cargo test` (unit tests), manual integration tests against local Synapse. No automated E2E tests in CI yet.

**Target Platform**: Linux x86_64 (CI/CD), macOS/Linux (developer machines). No mobile, no WASM.

**Project Type**: CLI binary (standalone executable)

**Performance Goals**: Subcommand startup <2s (dominated by core session restore + sync), `listen` latency <2s from send to display (same homeserver)

**Constraints**: Zero-config operation (no config files), all state in `./shadowlink_data/`, CWD-scoped sessions, non-zero exit on all errors, no panics

**Scale/Scope**: 11 subcommands, ~382 lines single-file, 1 developer/maintainer

| **V. Battery & Permission** | N/A | CLI is a developer tool, not a mobile app. Battery constraints do not apply. |
| **VI. CI Pipeline** | ✅ PASS | CLI CI: build, test, clippy, fmt, gitleaks. All gates green on `main`. |

**Gate status**: PASS with one documented gap (no Synapse integration tests in CI). Not a blocker for plan approval.

## Test Strategy

### Unit Tests

CLI has `cargo test` in CI. Current tests cover:
- Argument parsing (clap derive ensures this)
- Lat/lng validation

### Integration Tests (Manual)

Full manual test suite documented in `quickstart.md`:
1. Connect two users to local Synapse
2. Create room, invite, accept
3. Send/receive messages via `send` / `listen`
4. Share location, verify event
5. Disconnect, verify cleanup

### Future: Automated E2E Tests

Add Synapse service container to CLI CI workflow. Test scenarios:
- `connect` → verify `session.json` exists + valid
- `create-room` → verify room ID returned
- `send` + `listen` (two sessions) → verify message received
